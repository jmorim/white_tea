Tea is an important subject for me because it was the model crop on which my PhD thesis was based. At Tufts, we harvested tea grown in different seasons and elevations to see how its environment changed its flavor and health properties, all driven by its metabolism. Like many plants, tea uses chemical signals to communicate with its environment, since it can’t get up and walk away like an animal or even a microbe could. Evolution has driven myriad metabolic pathways to allow plants to defend themselves. This makes them particularly complex and challenging subjects of research, slowing studies on how crop qualities will change under different conditions, and exacerbated with the increasing threat and severity of climate change.

In this study, I discovered that two metabolites change strongly with storage time. They correspond to the differential features 607.2477_411 and 130.0507_69, named as m/z-RT pairs.

For this demo/tutorial, I looked for data collected from Camellia sinensis samples on MetaboLights1 and found one study in progress by Wei and Li.
This study aimed to investigate the sensory and chemical metabolites, both endogenous to the tea and exogenous from microbial contributions, over long-term storage.2 We will focus on the metabolomics data herein.

Data conversion

To read files with xcms,3,4 data need to be converted to .mzML format with MSConvert from the ProteoWizard package. Since I’m on Linux, I need a docker container to run this with a Windows image. On Windows, it can run natively. The code below detects the host environment and chooses the appropriate conversion command. The convertDtoMZML function from my LCMStools package was built for Agilent .d files, but it should work for .raw files too since the parameters aren’t too specific. Docker will prompt for a password for sudo access. By using the Docker image, you agree to certain vendor licenses for converting proprietary formats to .mzML.

Supplying readMsExperiment with metadata

Now that we’ve taken a first look at the data, I’m going to go back and supply the readMsExperiment function with the relevant sampleData information so we can use it downstream for peak grouping and statistical analyses. The original MsExperiment objects will be replaced so they don’t take up more memory.

Conveniently, we can build out sampleData from the assay data from MetaboLights. dplyr can filter the study.info table to split it into the positive and negative ion mode information, and stringr can extract the year value as a separate longitudinal variable. Sometimes, another package will override the filter function definition from dplyr, so it’s good to scope it to the right one. The study.info table also has columns with duplicate names that need to be renamed or removed to work with dplyr.

study.info.m = study.info
colnames(study.info.m) = make.unique(colnames(study.info.m))
study.info.m = study.info.m |>
  mutate(
    polarity = str_extract(`Sample Name`, 'POS|NEG'),
    year = str_extract(`Sample Name`, '\\_\\d+\\_') |> str_replace_all('\\_', '') |> as.numeric()
  )
pos.data = readMsExperiment(pos.files, 
  sampleData = study.info.m |> filter(polarity == 'POS'))

Peak picking

The next step is to get some starting parameters for peak picking. First, we’ll zoom in on the early and late parts to refine the retention time window on which peak picking will operate. xcmsVis provides ggplot version of xcms plots, and plotly enables interactions, although it’s slower than generating static plots, so I’m not going to incorporate it into my knitr output.

gplot(chromatogram(pos.data)) |> ggplotly()

Zooming in, chromatographic peaks start eluting at 25 s, which is the dead time. This is the time it takes for unretained sample to pass through the column and reach the detector. It can be determined based on the column geometry and flow rate. For example, in this study, the authors used a 2.1 × 100 mm column with a 5 mm guard column (notice the discrepancy in the study.info and the Chromatography section of the protocol info), but didn’t specify the flow rate. These columns are usually run at 0.1 – 0.2 mL/min. Let’s check using the observed dead time. Column volume can be estimated, depending on the porosity, as 0.7 × π r2 × L, in μL, where r is the column radius (half the column diameter noted above) and L is the column length. Note this ignores any dead volume contributions from extra-column tubing.

dead.time = 25
col.vol = 0.7 * pi * (2.1/2)^2 * 105
est.flow.rate = 60 * col.vol / dead.time
print(est.flow.rate)
## [1] 610.9801

I was a bit off. It looks like they used a flow rate of 0.500 – 0.600 mL/min, but that’s not unheard of in ultra-high performance liquid chromatography (UHPLC).

At the other end of the chromatograms, it looks like a diverter valve was used to redirect the stream when the gradient was done, and I don’t see any artifacts, so the retention window for peak picking will be c(25, 480). Since the same chromatographic method was used, this window should work for the negative mode data too.

Before delving into peak picking, we need to set up colors for plotting. I use colorspace to generate my palettes. For longitudinal studies, a sequential palette is appropriate. Simple A versus B studies can use diverging palettes.

Now to peak picking.

Manually looking at the plotly TIC, the biggest peak (between 207 and 223 s, probably caffeine) had a peakwidth of about 16 seconds, so I’ll round up to 20 for the max peakwidth. A smaller peak had a peakwidth of around 8 seconds, so I’ll round down to 5 for the min peakwidth. The ppm parameter can usually be estimated by the instrument type. For TOFs and Q-TOFs, I’ve had to set this between 20 – 80 ppm for good peak detection. Orbitraps are much more accurate on the m/z domain, so 10 ppm should be sufficient.

The snr parameter won’t be touched because it’s shown to not be particularly useful.5

Peak grouping/correspondence

Each chromPeak is specific to a single sample. They need to be grouped across samples to compare. Tracking peaks across samples is called “correspondence” in xcms.

Note, usually people perform retention time alignment before correspondence. In my opinion, if you need to align retention times, your chromatography is not robust enough and you’re not generating good data. Sometimes this is the case when sequences don’t use at least a few equlibration injections at the beginning. This particular study looks well-behaved.

Using the final results

We now basically have a table of feature intensities grouped across samples, which proxies for metabolite abundances. We can use this for downstream statistical analyses to find what are the important features that change with storage time. This is a lot less work than trying to identify every single peak and then finding out most of them weren’t important.

I usually start with a principal component analysis (PCA) to get an overview of the data.

Notice that some of the features are actually redundant, or degenerate, meaning they originate from the same compound or metabolite. In particular, the top left and middle feature show the distinct peak shape of isomers coeluting, and they elute at the same retention time. The middle right and bottom middle features also overlap. Another way to reduce the dataset is to group these degenerate features into compounds through the CAMERA package, which groups coeluting features into “pseudospectra.” However, this package hasn’t been updated to work with the new xcms interfaces introduced with version 3.0, so I haven’t touched it in a while. Another, more recent, package is MsFeatures by Johannes Rainer. This one looks promising, and since it’s so new I haven’t gotten to implement it yet. But the ideas behind it sound strong, as it groups features by how well they correlate in the retention time dimension.

I digress. My point is that a feature is not a compound, but once we get from features to compounds, data size can be reduced immensely.

Quantitatively, FT0396 just about doubled over 11 years, and FT4766 decreased to about 20% of its original abundance. This is a pretty big change, and it would be interesting to see what these compounds are. We can try to identify them by looking up their m/z values in metabolite databases.
